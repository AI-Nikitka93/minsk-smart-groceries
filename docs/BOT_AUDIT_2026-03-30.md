# Bot Audit — 2026-03-30

## Verdict

`NO-GO` for the intended product promise.

The bot is deployed and can respond, but it does not yet behave like a reliable personal grocery assistant for Minsk. The current implementation is a brittle hybrid of heuristics, planner prompts, and weak catalog grounding. It produces user-visible failures in profile capture, "where cheaper" queries, and basket quality.

## Evidence Summary

- Live user logs show:
  - `у меня диабет и бюджет 30 рублей` returned `Я пока не нашёл такой товар в базе...`
  - `где дешевле купить масло` returned irrelevant "butter-adjacent" products like croissant on butter and chicken steak with butter
  - `собери корзину на 3 дня` produced a nonsensical basket with sweets and sauces for a diabetic profile
- Local DB probes on 2026-03-30 show sparse or irrelevant exact coverage:
  - `масло` -> `1` available row
  - `молоко` -> `0`
  - `торт` -> `0`
  - `гречка` -> `0`
- Synthetic webhook checks confirm transport works, but not product quality.

## Root Cause

The current bot tries to solve product semantics, personalized planning, health reasoning, exact shopping retrieval, and answer composition in one Worker without a strict tool-contract layer or strong catalog normalization. It is halfway between:

- deterministic search bot,
- LLM planner,
- shopping assistant,
- dietary advisor.

That hybrid state is the main reason it feels "fake-smart" in user chat.

## What Must Change

### Phase A — Stabilize
1. Split profile update from search execution.
2. Add exact-match gating for `find_cheapest`.
3. Add safe send fallback: `Markdown -> plain text retry`.
4. Stop returning loose matches as exact shopping answers.

### Phase B — Real Tool-Orchestrated Assistant
1. Introduce explicit tools:
   - `save_user_profile`
   - `search_exact_products`
   - `search_similar_products`
   - `find_cheapest_offer`
   - `build_budget_basket`
   - `analyze_composition`
2. Give planner a strict per-action contract validator.
3. Add follow-up questions when the planner lacks enough product anchors.

### Phase C — Data Layer
1. Improve parser/category coverage for staple products.
2. Add commodity normalization:
   - масло
   - молоко
   - гречка
   - торт
   - хлеб
   - яйца
3. Add exact commodity aliases and category read-models.

## Recommended Next Milestone

`Milestone: Stabilization Sprint`

Definition:
- Profile messages always get a confirmation response.
- `where cheaper` answers only return exact or clearly labeled approximate matches.
- Basket flow refuses bad baskets and asks a follow-up instead.
- Telegram send path retries safely on parse-mode failures.

## Release Recommendation

`NO-GO`

The bot should not yet be presented as a best-in-segment personal grocery assistant for Belarus. It is a promising prototype with live infrastructure, but the shopping intelligence and reliability layers are not yet at product level.
